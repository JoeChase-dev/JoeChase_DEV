## 8.1 GIL 全局解释器锁

### GIL 是什么？

```
GIL = Global Interpreter Lock = CPython 的全局互斥锁

核心规则: 同一时刻，只有一个线程能执行 Python 字节码
```

### 为什么有 GIL？

1. **CPython 的内存管理**（引用计数）不是线程安全的
2. 大量 C 扩展库依赖 GIL 来保证安全
3. 历史遗留问题（从 Python 1 时代就有了）

### GIL 的影响

| 任务类型 | 多线程效果 | 原因 |
|---------|----------|------|
| **IO 密集型** (网络/文件/数据库) | ✅ **有效加速** | IO 等待时 GIL 会释放，其他线程可以运行 |
| **CPU 密集型** (计算/图像处理) | ❌ **不能并行** | 同一时间只有一个线程在执行 CPU 代码 |

### 实测：CPU 密集型多线程无效

```python
import threading, time


def cpu_bound(n):
    """CPU 密集型任务"""
    while n > 0:
        n -= 1


# 单线程基准
start = time.time()
cpu_bound(100_000_000)    # 1 亿次循环
single_time = time.time() - start
print(f'单线程耗时: {single_time:.2f}s')

# 双线程（期望加速 2x？）
t1 = threading.Thread(target=cpu_bound, args=(50_000_000,))
t2 = threading.Thread(target=cpu_bound, args=(50_000_000,))
s = time.time()
t1.start(); t2.start()
t1.join(); t2.join()
dual_thread_time = time.time() - s
print(f'双线程耗时: {dual_thread_time:.2f}s')
print(f'加速比: {single_time/dual_thread_time:.2f}x (理想应该是 2.00x)')
# 实际接近 1.00x → 没有加速! 因为 GIL 让同一时间只有一个线程跑!
```

> 📌 **结论**: CPU 密集型任务用**多进程**替代多线程！

## 8.2 多线程 (`threading`)

### 8.2.1 创建与启动

```python
import threading, time

def worker(name, seconds):
    print(f'[{name}] 开始工作...')
    time.sleep(seconds)
    print(f'[{name}] 完成! 耗时 {seconds}s')


# 创建线程对象
t1 = threading.Thread(target=worker, args=('线程A', 2))
t2 = threading.Thread(target=worker, args=('线程B', 1))

# 启动（非阻塞!）
t1.start()
t2.start()

print('[主线程] 继续做其他事...')

# 等待子线程完成
t1.join()
t2.join()
print('[主线程] 所有工作完成')
```

### 8.2.2 `Lock` —— 解决竞态条件 (Race Condition) ⚠️

#### 问题演示

```python
counter = 0

def unsafe_increment():
    global counter
    for _ in range(100000):
        counter += 1     # 这不是原子操作！读→改→写 三步


threads = [threading.Thread(target=unsafe_increment) for _ in range(10)]
for t in threads:
    t.start()
for t in threads:
    t.join()
    
print(f'预期: 1000000, 实际: {counter}')
# 结果可能 < 1000000! 因为多个线程同时读到相同的值!
```

#### 解决方案：加锁

```python
lock = threading.Lock()

def safe_increment():
    global counter
    for _ in range(100000):
        with lock:                    # 自动获取+释放锁
            counter += 1             # 这三步变成"原子操作"


counter = 0
threads = [threading.Thread(target=safe_increment) for _ in range(10)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(f'结果: {counter}')  # 1000000 ✓ 正确!
```

#### Lock vs RLock 对比

| 特性 | `Lock` (普通锁) | `RLock` (可重入锁) |
|------|:--------------:|:-----------------:|
| 同一线程重复 acquire | ❌ **死锁!** | ✅ 允许（内部计数+1）|
| 计数机制 | 无 | 有，acquire +1，release -1 |
| 适用场景 | 简单互斥 | **递归/嵌套加锁** |

> 📌 **选择原则**: 不确定是否会有重入 → **优先用 RLock**

### 8.2.3 `Condition` —— 生产者-消费者模型

> **⚠️ 关键坑：`wait()` 必须放在 `while` 循环中，不能用 `if`！**

```python
from threading import Thread, Condition
import time, random


class BoundedBuffer:
    """有界缓冲区 — 生产者-消费者模型的经典实现"""

    def __init__(self, capacity):
        self.capacity = capacity
        self.buffer = []
        self.cond = Condition()      # 条件变量

    def put(self, item):
        """生产者放入数据"""
        with self.cond:
            # ⚠️ 必须用 while! 防止虚假唤醒 (Spurious Wakeup)
            while len(self.buffer) >= self.capacity:
                print(f'[生产者] 缓冲区满，等待消费...')
                self.cond.wait()       # 等待消费者通知

            self.buffer.append(item)
            print(f'[生产者] 放入 {item} | 当前: {self.buffer}')
            self.cond.notify_all()    # 通知所有等待的消费者

    def get(self):
        """消费者取出数据"""
        with self.cond:
            while not self.buffer:   # ⚠️ 必须用 while!
                print(f'[消费者] 缓冲区空，等待生产...')
                self.cond.wait()

            item = self.buffer.pop(0)
            print(f'[消费者] 取出 {item} | 当前: {self.buffer}')
            self.cond.notify_all()    # 通知所有等待的生产者
            return item


buf = BoundedBuffer(capacity=3)


def producer(name, items):
    """生产者"""
    for item in items:
        time.sleep(random.uniform(0.05, 0.2))
        buf.put(f'{name}-{item}')


def consumer(name, count):
    """消费者"""
    for _ in range(count):
        time.sleep(random.uniform(0.1, 0.3))
        buf.get()


# 启动 2 个生产者和 2 个消费者
threads = [
    Thread(target=producer, args=('P1', list(range(4)))),
    Thread(target=producer, args=('P2', list(range(4)))),
    Thread(target=consumer, args=('C1', 5)),
    Thread(target=consumer, args=('C2', 3)),
]
for t in threads:
    t.start()
for t in threads:
    t.join()
print('\n✓ 生产者-消费者模型演示完毕!')
```

> 📌 **为什么用 `while` 不用 `if`？**
>
> ```
> 虚假唤醒 (Spurious Wakeup): 
> 操作系统有时会在没有 notify 的情况下"误唤醒"等待中的线程。
> 如果用 if: 只检查一次 → 可能拿到空数据或写入满缓冲区 → BUG!
> 如果用 while: 每次被唤醒都重新检查条件 → 安全!
> ```

### 8.2.4 `Semaphore` —— 限制并发数

```python
from threading import Semaphore, Thread
import time

sem = Semaphore(3)  # 同时最多允许 3 个连接


def access_resource(name):
    with sem:                     # 获取信号量（如果已满则阻塞等待）
        print(f'{name} 获得访问权限 ✓')
        time.sleep(1)
        print(f'{name} 释放权限')


threads = [Thread(target=access_resource, args=(f'T-{i}',)) for i in range(6)]
for t in threads:
    t.start()
for t in threads:
    t.join()

# 输出示例（同时只有 3 个在运行）:
# T-0 获得访问权 ✓
# T-1 获得访问权 ✓
# T-2 获得访问权 ✓          ← 第4个要等前面的释放
# ... 1秒后 ...
# T-0 释放权限
# T-3 获得访问权 ✓         ← 第4个终于进来了
```

### 8.2.5 `ThreadPoolExecutor` —— 线程池（推荐方式）

```python
from concurrent.futures import ThreadPoolExecutor, as_completed
import time, random


def task(name, duration):
    time.sleep(duration)
    return f'{name} 完成 ({duration}s)'


# 创建线程池（最多 3 个工作线程）
with ThreadPoolExecutor(max_workers=3) as pool:
    futures = []
    for i in range(6):
        future = pool.submit(task, f'任务-{i}', random.uniform(0.1, 0.5))
        futures.append(future)

    # as_completed: 谁先完成先取谁（不按提交顺序）
    for future in as_completed(futures):
        print(future.result())
```

> 📌 **线程池的优势**：
> - 避免频繁创建/销毁线程的开销
> - 控制最大并发数（防止资源耗尽）
> - 提供 `map()` / `submit()` / `as_completed()` 等便捷 API

## 8.3 多进程 (`multiprocessing`)

> **绕过 GIL，实现真正的多核并行！**

```python
from multiprocessing import Pool, cpu_count
import time, os


def heavy_computation(n):
    """CPU 密集型任务"""
    result = 0
    for _ in range(n * 500_000):
        result += 1
    return os.getpid(), result


print(f'CPU 核心数: {cpu_count()}')

# 单进程基准
start = time.time()
results_single = [heavy_computation(i) for i in range(4)]
single_t = time.time() - start
print(f'单进程: {single_t:.2f}s')

# 多进程并行（利用多核!）
start = time.time()
with Pool(processes=4) as pool:           # 4 个 worker 进程
    results_multi = pool.map(heavy_computation, range(4))
multi_t = time.time() - start
print(f'多进程(4核): {multi_t:.2f}s')
print(f'加速比: {single_t/multi_t:.2f}x')   # 接近 4x!

for pid, res in results_multi:
    print(f'  PID={pid}, result={res}')
```

### 进程间通信 (IPC) —— `Queue`

> **关键区别**: 进程之间内存独立！不能直接共享变量！

```python
from multiprocessing import Process, Queue
import time


def producer(queue):
    for i in range(5):
        queue.put(f'item-{i}')
        time.sleep(0.1)
    queue.put(None)              # 结束信号


def consumer(queue):
    while True:
        item = queue.get()
        if item is None:         # 收到结束信号
            break
        print(f'消费: {item}')


q = Queue()                      # 进程安全队列
p = Process(target=producer, args=(q,))
c = Process(target=consumer, args=(q,))
p.start(); c.start()
p.join(); c.join()
print('✓ IPC 演示完毕')
```

## 8.4 异步 IO (`async` / `await`) ⭐

> **单线程 + 事件循环** —— 通过协作式调度实现高并发。

### 基础用法

```python
import asyncio


async def say_after(delay, msg):
    """
    async def: 定义协程函数（调用后返回协程对象，不会立即执行）
    await:     暂停当前协程，让出控制权给事件循环去执行其他协程
    """
    await asyncio.sleep(delay)   # 协作式让出控制权（非阻塞!)
    print(msg)


async def main():
    print('开始')

    # 创建任务（但不立即执行，加入事件循环的调度队列）
    task1 = asyncio.create_task(say_after(2, 'hello'))   # 2秒后输出
    task2 = asyncio.create_task(say_after(1, 'world'))    # 1秒后输出

    await task1    # 等待 task1 完成
    await task2    # 等待 task2 完成

    print('结束')


asyncio.run(main())
# 输出顺序:
# 开始
# world      (1秒后先出来)
# hello      (2秒后才出来)
# 结束
# 总耗时约 2 秒 (不是 3 秒!) — 因为两个任务是并发执行的!
```

### async/await 的本质

```
同步代码 (一个接一个):
  sleep(2) → 输出 'hello' → sleep(1) → 输出 'world'
  总时间: 3 秒

异步代码 (交错执行):
  任务A: sleep(2) ............→ 输出 'hello'
  任务B:       .sleep(1). → 输出 'world'
  总时间: 2 秒 (两个任务的等待时间是重叠的!)
  
核心差异:
- 多线程: 抢占式调度（操作系统强制切换）需要锁保护
- 异步IO:  协作式调度（主动 yield 让出）不需要锁!
```

### aiohttp —— 异步 HTTP 实战

```python
# pip install aiohttp
import asyncio
import aiohttp


async def fetch(session, url):
    """异步 HTTP GET 请求"""
    async with session.get(url, timeout=aiohttp.ClientTimeout(total=5)) as resp:
        text = await resp.text(200)
        return url, resp.status, len(text)


async def main():
    urls = ['https://www.baidu.com', 'https://www.bing.com']
    
    async with aiohttp.ClientSession() as session:
        tasks = [fetch(session, url) for url in urls]
        results = await asyncio.gather(*tasks)  # 并发执行所有请求
        
        for url, status, length in results:
            print(f'{url[:20]:20s} → status={status}, body={length} bytes')


asyncio.run(main())
```

## 8.5 三种并发方式选型指南

### 一览表

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **多线程** `threading` | 少量 IO 任务（< 几十个）| 简单易用、轻量 | GIL 限制 CPU 性能 |
| **多进程** `multiprocessing` | CPU 密集型任务 | **真正多核并行** | 进程开销大、IPC 复杂 |
| **异步IO** `async/await` | 高并发网络请求（成千上万）| **极高的并发效率** | 只能用异步代码 |

### 快速选择流程图

```
你的任务是什么?
│
├─ CPU密集型 (计算/图像处理)
│   └──→ 用 multiprocessing 多进程
│
├─ IO密集型
│   ├─ 数量少 (< ~50) 且逻辑简单
│   │   └──→ 用 threading 多线程
│   │
│   └─ 数量大 (> ~50) 或 高并发需求
│       └──→ 用 async/await 异步IO
│
└─ 混合场景
    └──→ 线程池 + 异步组合使用
```

### 具体场景推荐

| 场景 | 推荐方案 |
|------|---------|
| 网络爬虫 | `asyncio` + `aiohttp` |
| 图像/视频批量处理 | `multiprocessing.Pool` |
| 文件 IO / 定时任务 | `threading` / `ThreadPoolExecutor` |
| Web 后端服务 | 框架自带异步支持 (FastAPI/Starlette) |
| GUI 应用后台任务 | `QThread`(Qt) / `threading` |

## 易错踩坑速查

| 坑 | 正确做法 |
|---|---------|
| `while` 条件中用 `if` 判断 Condition | **必须用 `while`** 防止虚假唤醒 |
| 死锁：A 等 B 的锁，B 也等 A | 固定加锁顺序 / 用 `RLock` / 超时机制 |
| 共享可变数据不加保护 | 用 `Lock` / `Queue` / 局部变量 |
| 在 `async` 函数里用阻塞 IO | 用 `aiofiles` / `aiohttp` / `run_in_executor()` |
| `asyncio.run()` 已有线程循环时报错 | 用 `await` 或 `loop.create_task()` |
| 子类化 `Thread` | **优先传 `target=`函数** 更简洁 |
| 忘记 `.join()` 等待线程 | 主程序退出后子线程可能被**强制终止** |
| 多进程中直接共享全局变量 | **不行！** 进程内存独立，用 `Queue`/`Manager` |
